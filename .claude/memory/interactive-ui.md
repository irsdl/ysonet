# Interactive UI (ysonet/Interactive/)

- 2026-07-22 - Screen-redraw convention: a screen must call `ConsoleCursor.ClearScreen()` once when it is entered or re-entered, then redraw in place with `MoveUp` for navigation within that screen; a sub-screen (e.g. an axis checklist) clears on its own entry and the parent clears again on return, wiping it - never append a screen beneath the previous one. - `MoveUp` only redraws in place within a single screen, so a parent -> child -> parent transition without a clear leaves both drawn and the menu appears twice on a real console (the "menu repeats down the screen" stacking bug; fixed in `CategoryFilter`; canonical note on the `Menu` class comment; full detail in `docs/ARCHITECTURE.md` section 4.1).
- 2026-07-22 - A redirected-console test cannot catch a real-console redraw/stacking bug; write the regression test against the `VirtualTerminal` harness (real cursor control) and assert the screen title never appears on two rows of one captured frame - see `CategoryFilterDoesNotStack`. - Under a redirected console `ClearScreen`/`MoveUp` are no-ops and output just appends, so the stacking is invisible to append-mode tests.
- 2026-07-22 - The `Picker` window shows only 12 rows, so an action/option row appended at the BOTTOM of a long list (e.g. the gadget list's `[ Filter by category... ]`) is off-screen until the user types or scrolls. Make it discoverable another way: a hint in the picker title, a footer/legend key (the live columns add `Ctrl+F`), and rely on type-to-filter surfacing it (typing "filter" narrows to that row). - Bottom placement keeps the first real item as the default selection (good for the common Enter-Enter pick), but trades away default visibility of trailing rows.
